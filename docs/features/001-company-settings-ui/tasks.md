# Tasks: Company Settings Management, UI Simplification, and Invoice Flexibility

## Overview

This document breaks down feature `001-company-settings-ui` into actionable implementation
tasks. The feature delivers three coordinated improvements:

1. **Company profile centralisation** — move Firmenprofil editing from wizard Screen 1 to
   the "Einstellungen" page; Screen 1 becomes a read-only summary.
2. **Single upload action per field** — remove the duplicate "Rechnung hochladen" button
   that appears when both `UploadOCR` and `FieldDocumentZone` are rendered for the same field.
3. **Multiple invoices per category** — replace the strict one-document-per-(fieldKey, year)
   limit with an append-based document list.

Reference documents: `specification.md`, `architecture.md`, ADRs 001–004, `test-plan.md`.

---

## Tasks

### Task 1: Remove `@@unique([fieldKey, year])` constraint from `FieldDocument` (Prisma migration)

**Priority:** High

**Description:**
The `FieldDocument` model currently enforces a unique index on `(fieldKey, year)`, limiting
each field to one document. Remove this constraint via a Prisma migration so that multiple
documents can coexist for the same `(fieldKey, year)` pair.

**Files to modify:**
- `prisma/schema.prisma` — remove `@@unique([fieldKey, year])` line from `FieldDocument`
- `prisma/migrations/<timestamp>_drop_fielddocument_unique/` — auto-generated by Prisma

**Acceptance Criteria:**
- [ ] `prisma/schema.prisma` no longer contains `@@unique([fieldKey, year])` on `FieldDocument`
- [ ] Running `npx prisma migrate dev --name drop_fielddocument_unique` succeeds without errors
- [ ] Existing `FieldDocument` rows are preserved unchanged after the migration (non-destructive)
- [ ] `prisma generate` completes successfully (Prisma Client no longer exposes `fieldKey_year` unique accessor)
- [ ] The migration SQL drops the unique index and does not touch data rows

**Dependencies:** None

**Notes:**
After this migration, the `GET /api/field-documents` route's `findUnique` call will break
(unique accessor removed). Task 2 must be completed promptly. Also: the `POST` route's
`upsert` will break. Task 3 must also be completed before the app can run.

- [ ] The route still validates that `fieldKey` and `year` query params are present, returning `400` if missing
- [ ] `prisma.fieldDocument.findUnique` and the `fieldKey_year` unique accessor are completely removed from the route

**Dependencies:** Task 1

### Task 3: Update `POST /api/field-documents` to use `create` with a timestamp-prefixed filename

**File to modify:**
- `src/app/api/field-documents/route.ts`

- [ ] Stored filename format is `${Date.now()}_${base}.${ext}` (e.g., `1720000000000_rechnung.pdf`)
- [ ] The relative file path stored in `FieldDocument.filePath` reflects the new timestamped filename
- [ ] Two consecutive `POST` requests with the same `fieldKey`/`year`/`filename` create two

- [ ] File sanitisation logic (alphanumeric, hyphens, underscores only; single extension) is
  preserved for the base name portion
- [ ] `400` is still returned if `file`, `fieldKey`, or `year` are missing

### Task 4: Add `DELETE /api/field-documents/[id]` endpoint

**Priority:** High

**Description:**
Create a new API route that deletes a single `FieldDocument` by its database `id`. The file
path is retrieved from the DB record (not from the request) to prevent path-traversal attacks.
The physical file is deleted from the filesystem before the DB record is removed.

**File to create:**
- `src/app/api/field-documents/[id]/route.ts`

**Acceptance Criteria:**
- [ ] `DELETE /api/field-documents/42` returns `204 No Content` on success
- [ ] The physical file at `public/${doc.filePath}` is deleted from the filesystem
- [ ] If the physical file does not exist on disk (already deleted), the error is caught, logged, and the DB record is still deleted — returning `204`
- [ ] If no `FieldDocument` with the given `id` exists, the route returns `404`
- [ ] The file path used to locate the file on disk comes exclusively from the DB record, **never** from the request body or URL parameters (security requirement per architecture doc)
- [ ] The `id` parameter is parsed as an integer; non-numeric values return `400`

**Dependencies:** Task 1

**Notes:**
This endpoint is consumed by `FieldDocumentZone`'s "Entfernen" button (Task 6).

### Task 10: Refactor `FieldDocumentZone` for multi-document support

**Priority:** High

   "Rechnung hochladen" button in the empty state

**File to modify:**

  to `[]`

  array; re-runs when `refreshKey` changes (add `refreshKey` to effect dependency array)
- [ ] When the document list is non-empty, each document is rendered as a row showing:
  filename, upload date (formatted as `DD.MM.YYYY`), "Ansehen" link (`href={doc.filePath}`,
  `target="_blank"`), and "Entfernen" button
- [ ] Clicking "Entfernen" calls `DELETE /api/field-documents/{doc.id}` and removes the
  document from the displayed list on success
- [ ] Deleting one document does not affect others in the list
- [ ] Empty state (no documents) shows "📄 Kein Dokument hochgeladen" placeholder

- [ ] "Weitere Rechnung hinzufügen" button is visible when `suppressInitialUpload={true}` OR

- [ ] When `docs.length >= 20`, a soft warning message is shown (e.g., yellow/amber notice in German)
- [ ] Upload action (from both initial button and "Weitere Rechnung hinzufügen") calls

- [ ] Upload errors are shown inline without removing existing documents
- [ ] The "Ersetzen" button from the single-doc implementation is **removed**
