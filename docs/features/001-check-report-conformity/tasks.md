# Tasks: Check Report Conformity (Feature 001)

## Overview

This feature introduces a pre-report conformity check that validates emission data against the GHG Protocol Corporate Standard and CSRD supply-chain data-quality expectations. A badge and panel surface the results to users on the dashboard before PDF download. Wizard screens gain double-counting detection and missing-category prompts. PDFs are enriched with a data-completeness table, a conformity footer, and a Scope 3 completeness section.

Reference: `docs/features/001-check-report-conformity/specification.md` and `docs/adr/001-conformity-check-architecture.md`.

### Task 2: Core Logic — Implement `src/lib/conformity.ts`

**Priority:** High

**Description:**
Create the pure synchronous conformity engine. This is the single source of truth for all check rules and is used by the dashboard, API route, and PDF generation.

**Files to create:**
- `src/lib/conformity.ts`

**Key implementation details:**
- Export types: `ConformityIssueSeverity`, `ConformityIssue`, `ConformityStatus`, `CategoryStatus`, `ConformityCheckResult`, `ConformityCheckInput`
- Export `runConformityChecks(input: ConformityCheckInput): ConformityCheckResult` — implements GHG-01 through GHG-05, GHG-W01 through GHG-W04, and CSRD-01 through CSRD-04 as described in the specification
- Export `detectDoubleCounting(entries): ConformityIssue[]` — implements DC-01 and DC-02
- Export `getMissingCategoryNotices(branche, entries): ConformityIssue[]` — uses the Branche → expected categories mapping
- Export `BRANCHE_EXPECTED_CATEGORIES` constant with the mapping table from the spec
- `status` is derived: any `errors` → `'nicht-konform'`; no errors but `warnings` → `'hinweise'`; neither → `'konform'`
- All `titleDe` and `descriptionDe` strings must be in German
- Each `ConformityIssue` includes a `wizardScreen` number (1–7) where applicable for navigation links
- Function is **synchronous** and **has no I/O side effects** (no Prisma, no fetch)

**GHG check rules summary (see spec for full details):**
- GHG-01: ≥ 1 confirmed Scope 1 entry with qty > 0
- GHG-02: ≥ 1 STROM or FERNWAERME entry with qty > 0
- GHG-03: year ≥ 2020 and is a 4-digit integer
- GHG-04: profile non-null, `firmenname`/`standort`/`branche` non-empty, `mitarbeiter` ≥ 1
- GHG-05: sum of all scope totals ≥ 0
- GHG-W01: ≥ 1 Scope 3 entry OR `scope3Acknowledged === true`
- GHG-W02: at least one expected category per branche is captured
- GHG-W03: < 50% of non-zero-expected categories have data
- GHG-W04: vehicle-branche (KFZ_WERKSTATT, BAUGEWERBE, SHK, ELEKTROHANDWERK) with Scope 1 total < 0.5 t
- CSRD-01 through CSRD-04: info notices for Scope 3 categories and Ökostrom flag

**Acceptance Criteria:**
- [ ] All 25 test cases in `test-plan.md` pass
- [ ] Function signature matches the `ConformityCheckInput` / `ConformityCheckResult` types in the ADR
- [ ] `detectDoubleCounting` and `getMissingCategoryNotices` are exported and usable independently
- [ ] All user-facing strings are in German
- [ ] No Prisma or HTTP imports in this file

**Dependencies:** Task 1 (for the `scope3Acknowledged` field type awareness)
