# ADR-003: Data Model for Year-Specific Emission Factors

## Status

Accepted

## Context

The `EmissionFactor` model in `prisma/schema.prisma` already supports year-specific
factor versioning via a `validYear` column and a `@@unique([key, validYear])` constraint:

```prisma
model EmissionFactor {
  id        Int      @id @default(autoincrement())
  key       String
  validYear Int
  factorKg  Float
  unit      String
  source    String   // e.g. "UBA 2024"
  scope     Scope
  createdAt DateTime @default(now())

  @@unique([key, validYear])
}
```

The `source` field is already used to store the origin label (e.g. `"UBA 2024"`).
The `factors.ts` library already implements year-fallback logic.

The new feature requires:
1. **German display label per factor** (for the methodology table and the settings UI):
   e.g., "Erdgas" for key `ERDGAS`, "Strom (Ökostrom)" for `STROM_OEKOSTROM`.
2. **Custom/official distinction** visible in the UI and methodology block.
3. **No silent cross-year contamination** — updating year 2024 factors must not affect
   2023 factors.

---

## Options Considered

### Option 1: Derive Labels from Existing `CATEGORY_LABELS` Mapping in Code
Use the static `CATEGORY_LABELS` / `MATERIAL_CATEGORY_LABELS` lookup in `src/types.ts`
to compute display labels at the API layer. No schema change.

**Pros**
- Zero migration — no DB schema change required.
- Labels stay consistent with the wizard screen labels (same source).

**Cons**
- Some factor keys don't match `EmissionCategory` exactly (e.g., `STROM_OEKOSTROM`,
  `PKW_BENZIN_KM`, `PKW_DIESEL_KM`) and would require a separate supplementary mapping.
- Materials factors (`KUPFER`, `STAHL`, etc.) exist in `MaterialCategory` but are
  stored in the same `EmissionFactor` table — a second lookup is needed.
- Any mismatch between factor keys and type keys silently produces an empty label.

### Option 2: Add `label` Column to `EmissionFactor` Table ✅ *Recommended*
Add an optional `label String?` field to the `EmissionFactor` model. The UBA reference
module (`uba-reference-data.ts`) populates this field when seeding or auto-filling. The
settings UI reads it directly from the DB.

**Pros**
- Factor records are self-contained — the API can return label without a separate
  type-mapping lookup.
- Handles edge-case keys (`STROM_OEKOSTROM`, refrigerant keys, material keys) without
  a parallel mapping table.
- Labels can be localized or adjusted per year (unlikely but possible).
- The label field is populated once by the seed/auto-fill and never requires user entry.

**Cons**
- Requires a Prisma migration (`ALTER TABLE`).
- Existing seeded records have no label (NULL) until re-seeded or migrated. The API
  must fall back to a static lookup for `null` labels during the transition.

### Option 3: Add a Separate `UbaFactorLabel` Table
A standalone table mapping `key → label` for lookup.

**Pros**
- Completely separates label management from factor versioning.

**Cons**
- Introduces a join for every factor query, disproportionate complexity.
- The label–key mapping doesn't change per year, making a separate table overkill.

### Option 4: Add a New `EmissionFactorSet` Table to Group Factors by Year
A header table (`EmissionFactorSet`) with `year`, `sourceLabel`, `isCustom`, and a
one-to-many relation to `EmissionFactor`.

**Pros**
- Allows setting a single source label for the entire year's factor set in one place.

**Cons**
- Significant schema change (new model + FK relation + migration).
- Adds complexity to factor queries (join required).
- The existing `source` field per-row approach already supports per-factor override
  labelling, which is more granular and already in use.

---

## Decision

**Option 2 — Add `label String?` to the existing `EmissionFactor` model**, using the
existing `source` field to track official vs. custom origin (no new field for that).

**Custom/official distinction** is handled by the `source` field convention:
- `"UBA {year}"` = official UBA value (loaded by auto-fill or seed).
- `"Benutzerdefiniert {year}"` = manually overridden value.

When a user edits a factor inline, the API updates both `factorKg` and `source` for
that record.

---

## Rationale

Adding a `label` column is the smallest schema change that makes factor records
self-contained for display. The `source` field convention is already established and
semantically correct for the custom/official distinction — no new field is needed.
Cross-year isolation is already guaranteed by the `@@unique([key, validYear])`
constraint and the year-scoped upsert logic in the auto-fill endpoint.

The migration is safe: the new column is nullable, so existing rows are unaffected.
The seed script is updated to populate labels; the auto-fill endpoint also writes labels.

---

## Consequences

### Positive
- Factor records are self-describing (key + label + value + unit + scope + source).
- No code-side label mapping needed in API responses.
- Null-safe fallback to static mapping preserves backward compatibility.

### Negative
- One Prisma migration required: `npx prisma migrate dev --name add_label_to_emission_factor`.
- Existing seeded records have `label = null` until the seed is re-run. The API
  response should fall back to a static `FACTOR_LABELS` constant for null labels.

---

## Implementation Notes

**Schema change** (`prisma/schema.prisma`):
```prisma
model EmissionFactor {
  id        Int      @id @default(autoincrement())
  key       String
  validYear Int
  factorKg  Float
  unit      String
  source    String
  label     String?  // German display name, e.g. "Erdgas", "Strom (Ökostrom)"
  scope     Scope
  createdAt DateTime @default(now())

  @@unique([key, validYear])
}
```

**Prisma migration**: `npx prisma migrate dev --name add_label_to_emission_factor`

**`src/lib/uba-reference-data.ts`** — each `UbaFactor` entry includes `label`.

**`prisma/seed.ts`** — refactored to use `UBA_REFERENCE_DATA` (see ADR-001); labels
are written at seed time.

**`POST /api/emission-factors/uba-fill`** — sets `label` when writing factors.

**`PUT /api/emission-factors/[id]`** — updates `factorKg` and `source`
(`"Benutzerdefiniert {year}"`); does NOT overwrite `label` (label is stable).

**Static fallback** (`src/lib/factor-labels.ts`): A `FACTOR_LABELS: Record<string, string>`
constant provides a German label for any key where the DB `label` field is null.
This ensures the UI works even against a DB that has not been re-seeded.

**Year isolation** is automatic: the `@@unique([key, validYear])` constraint prevents
any operation from writing to a different year's row. The auto-fill endpoint uses
`upsert` with `where: { key_validYear: { key, validYear: year } }`.
