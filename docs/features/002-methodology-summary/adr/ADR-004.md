# ADR-004: API Design for UBA Emission Factor Management

## Status

Accepted

## Context

The new "Emissionsfaktoren" settings section requires API endpoints for:
1. Listing emission factors for a selected year (read).
2. Listing which years have factor data in the DB (for the year selector dropdown).
3. Updating individual factor values inline (write with source override tracking).
4. Auto-filling all factors for a year with official UBA values (bulk write).
5. Serving methodology data for the UI collapsible component.

The existing API surface does **not** include any routes for managing `EmissionFactor`
records. The `factors.ts` library has `getEmissionFactor` and `getAllFactorsForYear`
for read-only use by the calculation engine.

All routes must follow the established Next.js App Router pattern: file-based routes
under `src/app/api/`, Node.js runtime, TypeScript, Prisma ORM.

---

## Options Considered

### Option 1: Single Resource Route with Query Parameters
All emission-factor operations under a single `/api/emission-factors` route with HTTP
method + query params driving the action (e.g., `?action=uba-fill&year=2024`).

**Pros**
- One route file.

**Cons**
- Violates REST conventions (actions should not be query params).
- Hard to differentiate intentions for code readers (and for the Next.js router which
  uses filename-based route segments).
- Mixes read and write concerns in one handler.

### Option 2: Standard RESTful Routes with Action Sub-resource ✅ *Recommended*
- `GET  /api/emission-factors?year={year}` — list factors for a year
- `GET  /api/emission-factors/years` — list years with factor data
- `PUT  /api/emission-factors/[id]` — update a single factor (inline edit)
- `POST /api/emission-factors/uba-fill` — auto-fill UBA values for a year (action)
- `GET  /api/methodology?yearId={yearId}` — return `MethodologyData` for UI component

**Pros**
- Clear HTTP semantics: GET reads, PUT updates a specific resource, POST on a
  sub-resource (`uba-fill`) for the bulk action.
- Follows the established pattern in the codebase (e.g., `/api/entries`, `/api/report`).
- Easy to test each endpoint in isolation.
- `uba-fill` as a named sub-resource makes the action intent unambiguous.

**Cons**
- Multiple route files, but this is standard for Next.js App Router.

### Option 3: Server Actions Only
Implement factor management entirely through Next.js Server Actions (like `saveCompanyProfile`
in `src/lib/actions.ts`).

**Pros**
- Consistent with the `FirmenprofilSettings` component pattern.
- No explicit API route needed.

**Cons**
- Server Actions are not easily testable in isolation.
- The methodology endpoint must be a proper GET route anyway (fetched by a client component).
- The factor table is a data-heavy operation; REST routes are easier to paginate and
  cache in the future.
- Mixed patterns in a single feature would be confusing.

---

## Decision

**Option 2 — Standard RESTful Routes** with a named `uba-fill` action sub-resource.

---

## Rationale

RESTful routes are the established pattern for data-heavy read/write operations in this
codebase (`/api/entries`, `/api/report`, `/api/field-documents`). The `uba-fill` action
is a natural candidate for a POST to a sub-resource (it is a bulk mutation with
side-effects, not a plain CRUD operation). A separate `GET /api/methodology` route
keeps the methodology data fetching clean and independently cacheable.

---

## Consequences

### Positive
- Each endpoint has clear, focused responsibility.
- Easy to add pagination to `GET /api/emission-factors?year=` in the future.
- The `uba-fill` endpoint can return a summary (count of upserted factors) for the
  success toast.
- `GET /api/methodology` is reusable by any component that needs to display methodology
  data.

### Negative
- 5 new route files (manageable; consistent with existing API surface).

---

## Implementation Notes

### Route File Locations

```
src/app/api/emission-factors/route.ts          → GET (list factors for a year)
src/app/api/emission-factors/years/route.ts    → GET (list available years)
src/app/api/emission-factors/[id]/route.ts     → PUT (update single factor)
src/app/api/emission-factors/uba-fill/route.ts → POST (auto-fill official UBA values)
src/app/api/methodology/route.ts               → GET (methodology data for UI)
```

---

### `GET /api/emission-factors?year={year}`

Returns all `EmissionFactor` records for the given year, sorted by scope then key.

**Response** `200 OK`:
```json
[
  {
    "id": 1,
    "key": "ERDGAS",
    "label": "Erdgas",
    "factorKg": 2.0,
    "unit": "m³",
    "scope": "SCOPE1",
    "source": "UBA 2024",
    "validYear": 2024
  }
]
```

**Error** `400`: missing or invalid `year`.

---

### `GET /api/emission-factors/years`

Returns an array of years (integers) that have at least one `EmissionFactor` record in
the DB, sorted ascending. Used to populate the year selector in the settings UI.

Also includes which years have built-in UBA reference data available (from
`getUbaReferenceYears()` in `uba-reference-data.ts`), so the UI can enable/disable the
auto-fill button.

**Response** `200 OK`:
```json
{
  "dbYears": [2023, 2024],
  "ubaReferenceYears": [2023, 2024]
}
```

---

### `PUT /api/emission-factors/[id]`

Updates a single factor value. Sets `source` to `"Benutzerdefiniert {validYear}"` to
mark the factor as manually overridden.

**Request body**:
```json
{ "factorKg": 2.05 }
```

**Response** `200 OK`: the updated `EmissionFactor` record.

**Error** `400`: invalid `factorKg` (must be a finite number).
**Error** `404`: factor not found.

**Side-effect**: `source` is set to `"Benutzerdefiniert {validYear}"` automatically
when the stored value differs from the UBA reference value. The API does not need to
look up the reference — it always marks any inline edit as "Benutzerdefiniert".

---

### `POST /api/emission-factors/uba-fill`

Replaces all `EmissionFactor` records for the given year with the official UBA values
from `UBA_REFERENCE_DATA`. This is a **full replace** (all existing factors for the
year are overwritten, including any previously customized values — matching the
specification's explicit requirement).

**Request body**:
```json
{ "year": 2024 }
```

**Response** `200 OK`:
```json
{ "upsertedCount": 30 }
```

**Error** `400`: missing or unknown year (no UBA reference data for that year).

**Implementation**:
```typescript
import { UBA_REFERENCE_DATA } from '@/lib/uba-reference-data';

// For each factor in UBA_REFERENCE_DATA[year]:
await prisma.emissionFactor.upsert({
  where: { key_validYear: { key: factor.key, validYear: year } },
  update: { factorKg: factor.factorKg, source: factor.source, label: factor.label },
  create: { ...factor, validYear: year },
});
```

---

### `GET /api/methodology?yearId={yearId}`

Returns `MethodologyData` (see ADR-002) for the given reporting year. Used by the
`<MethodologySummary>` client component on the dashboard.

**Response** `200 OK`: a `MethodologyData` JSON object.

**Error** `400`: missing `yearId`.
**Error** `404`: reporting year not found.

**Runtime**: `nodejs` (same as the report route — this function may share logic with
the PDF generation path).

---

### Confirmation Dialog (Frontend)

The `POST /api/emission-factors/uba-fill` endpoint is destructive. The calling component
must show a confirmation dialog before invoking it:

> "Dies überschreibt alle bestehenden Faktoren für {Jahr} mit den offiziellen
> UBA-Werten. Bestehende Werte (auch manuell geänderte) werden ersetzt."

On success: toast `"UBA {Jahr} Faktoren wurden erfolgreich übernommen."`
On error: inline error message (do not discard existing factor display).

---

### Validation

All factor write endpoints validate `factorKg` as a finite number (positive or negative
— negative factors like `ABFALL_ALTMETALL` are valid). The `uba-fill` endpoint validates
that the requested year exists in `UBA_REFERENCE_DATA` before writing anything to the DB.
