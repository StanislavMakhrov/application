# ADR-001: UBA Reference Emission Factor Data — Storage and Sourcing Strategy

## Status

Accepted

## Context

The feature requires a one-click "UBA-Werte übernehmen" action that fills the emission
factor table for a given reporting year with official UBA values. This needs a reliable,
offline-capable source of the official UBA factor values for each known year (at minimum
2023 and 2024).

The application has the following constraints:
- **No external network calls**: Air-gapped / self-hosted Docker deployment; UBA
  website integration is explicitly out of scope.
- **UBA data must be bundled in the application**: Not fetched at runtime from an
  external service.
- **Year-specific factor sets**: Each year may have slightly different UBA values
  reflecting official annual updates.
- **Auto-fill API endpoint needs access at runtime**: The auto-fill endpoint
  (`POST /api/emission-factors/uba-fill`) must be able to read official values without
  executing a database query — the purpose is to *write* to the DB, not read from it.

The `prisma/seed.ts` currently seeds only **UBA 2024** factors. There is no mechanism to
add new years' factor sets beyond editing the seed script.

---

## Options Considered

### Option 1: Extend the Seed Script Only
Add UBA 2023 (and future years) directly into `prisma/seed.ts` alongside the existing
UBA 2024 block.

**Pros**
- No new file; uses the established pattern.
- DB remains the single source of truth.

**Cons**
- Seed script must be re-run to add new years — not triggered by a UI action.
- The auto-fill API endpoint would need a separate static source anyway (it cannot
  query the DB for the "official" values after user overrides have been applied).
- Mixes production seed data with reference catalog data in one file.

### Option 2: Static JSON Reference File (`prisma/uba-factors.json`)
Store official UBA values as a JSON file alongside the Prisma schema, imported by
both the seed script and the API route.

**Pros**
- Machine-readable, easy to diff in PRs.
- Decouples reference catalog from the seed script.

**Cons**
- JSON has no TypeScript type checking; schema drift is hard to catch.
- Requires a separate type definition to keep the API and seed in sync.

### Option 3: TypeScript Reference Module (`src/lib/uba-reference-data.ts`) ✅ *Recommended*
Define official UBA factor sets as a typed TypeScript constant in a dedicated module.
Each year is a key in a `Record<number, UbaFactorSet[]>`. The seed script imports this
module; the auto-fill API route also imports it directly.

**Pros**
- Full TypeScript type safety — a factor object missing a required field is a compile
  error.
- Single source of truth shared between seed and runtime API.
- Easy to add future years: append a new entry to the record, which is picked up by
  both the seed and the auto-fill endpoint without any other changes.
- Versioned in source control; changes are visible in PR diffs.
- No external network access; fully compatible with air-gapped deployment.

**Cons**
- Application bundle includes UBA reference data (negligible: ~30 factor rows per year).
- Not editable from a UI without re-deploying (by design — official UBA values should
  be immutable in source; only DB records are user-editable).

---

## Decision

**Option 3 — TypeScript Reference Module** (`src/lib/uba-reference-data.ts`).

---

## Rationale

The auto-fill API endpoint must compare against, and write, official UBA values at
request time. This rules out seed-script-only approaches. A static TypeScript module
provides type safety, compile-time validation, and a clear diff path for adding future
year data. The absence of network access is guaranteed by design — no external call is
possible.

---

## Consequences

### Positive
- Adding a new UBA year is a one-file change (add a new entry to `UBA_REFERENCE_DATA`)
  that is naturally reviewed in PRs.
- The seed script and auto-fill endpoint always use the same reference values.
- TypeScript compiler enforces that every factor object has all required fields.

### Negative
- Official UBA values for new years require a code deploy to be available in the app.
  (Acceptable: UBA annual data changes at a predictable, low cadence — once per year.)

---

## Implementation Notes

**New file:** `src/lib/uba-reference-data.ts`

```typescript
export interface UbaFactor {
  key: string;
  label: string;        // German display name
  factorKg: number;
  unit: string;
  source: string;       // e.g. "UBA 2024"
  scope: 'SCOPE1' | 'SCOPE2' | 'SCOPE3';
}

export const UBA_REFERENCE_DATA: Record<number, UbaFactor[]> = {
  2023: [ /* UBA 2023 factors — same keys as 2024, adjusted values */ ],
  2024: [ /* existing UBA 2024 factors from seed.ts, now with labels */ ],
};

/** Returns the years for which built-in UBA reference data exists. */
export function getUbaReferenceYears(): number[] {
  return Object.keys(UBA_REFERENCE_DATA).map(Number).sort();
}
```

**`prisma/seed.ts`** — refactor to import `UBA_REFERENCE_DATA` and iterate over it,
replacing the inline factor array.

**Auto-fill API** (`POST /api/emission-factors/uba-fill`) — imports `UBA_REFERENCE_DATA`,
looks up the requested year, upserts all factors into the DB.

**Coverage**: Ship UBA 2023 and UBA 2024 factor sets at minimum. The UBA 2023 values
differ slightly from UBA 2024 (mainly the `STROM` grid factor). The Developer should
populate exact 2023 values from the official UBA Emissionsfaktoren-Datenbericht.
