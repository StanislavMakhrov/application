# ADR-002: Methodology Block Assembly — Generation Strategy

## Status

Accepted

## Context

Every generated GHG Protocol and CSRD PDF report must include a "Methodik" section.
The same content must also appear in the web UI as a collapsible component, so users
can review the methodology without downloading the PDF.

The specification requires that the methodology block be **assembled automatically at
report-generation time from live database data** — not hardcoded and not manually
authored by the user.

The current `GHGReport.tsx` contains a static, hardcoded methodology paragraph that
references "UBA 2024" regardless of the actual year or factors used:

```tsx
<Text style={...}>
  Diese CO₂-Bilanz wurde nach dem GHG Protocol Corporate Standard erstellt. 
  Alle Emissionsfaktoren stammen aus dem Umweltbundesamt (UBA) Datenbericht 2024.
  ...
</Text>
```

This must be replaced with dynamic content assembled from real data.

The methodology block must include:
- Applied calculation standard
- Emission factor source + year (from the `source` field of `EmissionFactor` records)
- Included scopes (only scopes with at least one non-zero entry)
- Input-method breakdown (count of entries by `inputMethod`)
- Reporting boundary notes and exclusions (from `CompanyProfile`)
- Per-category factor table (factor key, German label, value, unit, scope)

---

## Options Considered

### Option 1: Assemble at Report Generation Time (on-demand) ✅ *Recommended*
A new service function `getMethodologyData(yearId)` queries the DB at report request
time and returns a structured `MethodologyData` object. The PDF component and UI
component both consume this structure.

**Pros**
- Always reflects current DB state — factor overrides, boundary note edits, and new
  entries are instantly reflected in the next generated report.
- No storage overhead — methodology data is derived, not stored.
- Consistent with the existing pattern for `getTotalCO2e()` (computed on-demand).
- Single source of truth: same `getMethodologyData` function used by both PDF and UI.

**Cons**
- Adds 2–3 additional DB queries per report generation (factors, entry counts by
  method, scopes with non-zero values). Given the small data sizes (< 30 factors,
  < 100 entries), this is negligible and stays well within the 3-second PDF SLA.

### Option 2: Store Methodology Block as Text in the `Report` Table
When a report is generated, serialize the methodology block and store it as JSON in
a `methodologySnapshot` column on the `Report` model.

**Pros**
- Historical reports have an immutable methodology snapshot tied to the exact state
  at generation time — useful for audit trails.

**Cons**
- Adds schema complexity and a migration.
- UI needs to choose whether to show "current" methodology or the last-generated
  snapshot — confusing for users who haven't generated a PDF yet.
- Snapshot staleness: if a user edits factor values after generating a PDF, the UI
  shows stale data from the snapshot, not the current DB state.
- Audit-log changes to factors (a deferred nice-to-have) would be a better solution
  for historical immutability than a snapshot column.

### Option 3: Cache Methodology Data with Invalidation
Compute `getMethodologyData` on first use and cache it in memory (or Redis), with
cache invalidation on factor or entry changes.

**Pros**
- Marginally faster for repeated report requests.

**Cons**
- Significant added complexity for a single-tenant, low-traffic application.
- Cache invalidation requires hooking into every write path that affects methodology
  data (factors, entries, profile).
- The data is small enough that recomputation is cheaper than maintaining a cache.

---

## Decision

**Option 1 — Assemble at report generation time** via a new `getMethodologyData(yearId)`
service function in `src/lib/`.

---

## Rationale

On-demand assembly is the simplest approach that satisfies all requirements. It is
consistent with the established pattern (`getTotalCO2e` also runs on every report
request), requires no schema changes, and ensures the methodology block never shows
stale data. The performance overhead (a few DB queries over a < 30 row table) is
negligible.

---

## Consequences

### Positive
- Methodology block is always accurate and up-to-date.
- No additional storage model needed.
- Same function serves both PDF generation and UI preview.

### Negative
- Report generation adds ~2–3 DB round-trips; acceptable given the small dataset.
- Historical audit of "what factors were in place when this PDF was generated" is not
  supported by this approach (deferred — the existing `AuditLog` mechanism covers
  factor changes if implemented in the future).

---

## Implementation Notes

**New service function:** `src/lib/methodology.ts`

```typescript
export interface MethodologyData {
  standard: string;                      // e.g. "GHG Protocol Corporate Standard"
  factorSourceLabel: string;             // e.g. "UBA 2024" or "Benutzerdefiniert 2024"
  factorYear: number;
  includedScopes: ('SCOPE1' | 'SCOPE2' | 'SCOPE3')[];
  inputMethodCounts: {
    manual: number;
    ocr: number;
    csv: number;
  };
  assumptions: string | null;            // reportingBoundaryNotes
  exclusions: string | null;
  factors: Array<{
    key: string;
    label: string;                       // German display name
    factorKg: number;
    unit: string;
    scope: string;
    source: string;                      // "UBA 2024" or "Benutzerdefiniert 2024"
  }>;
}

export async function getMethodologyData(yearId: number): Promise<MethodologyData> { ... }
```

**Scope inclusion rule**: Include a scope if it has at least one `EmissionEntry` with
`quantity ≠ 0` (positive or negative — recycling credits with negative quantities count
as participation in that scope). A scope where all entries have `quantity = 0` is
excluded.

**Factor source label rule**: 
- If **all** factors for the year have `source` matching `"UBA {year}"`, the label is
  `"UBA {year} Emissionsfaktoren"`.
- If **any** factor has a non-UBA source (i.e., manually overridden to
  `"Benutzerdefiniert {year}"`), the label becomes `"Benutzerdefiniert {year}"`.
- This answers Open Question 5 from the specification: the entire set is marked as
  "Benutzerdefiniert" when any factor has been overridden, for conservative auditability.

**Integration points**:
1. `src/app/api/report/route.ts` — call `getMethodologyData(yearId)` and pass the result
   as a prop to both `GHGReport` and `CSRDQuestionnaire`.
2. `src/components/reports/GHGReport.tsx` — replace the static methodology text with a
   `<MethodologySection data={methodologyData} />` PDF component.
3. `src/components/reports/CSRDQuestionnaire.tsx` — add a methodology section (as a
   dedicated section, not an appendix — see architecture.md for rationale).
4. New UI component `src/components/reports/MethodologySummary.tsx` — renders the
   same `MethodologyData` as an expandable web component for the dashboard.
5. Dashboard year card — adds a `<MethodologySummary yearId={yearId} />` client component
   that fetches `GET /api/methodology?yearId={yearId}` and renders collapsible content.
